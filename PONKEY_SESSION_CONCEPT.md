## 0. このドキュメントについて

PONKEY SESSION は「PONKEY A(人間)を弾くと、PONKEY B(AI)が音楽で返事をし、PONKEY C(AI)がドラムをつける」三体合奏アプリである。本書はその**設計思想・アーキテクチャ・各機能の「なぜ」**を網羅する。実装の行単位の詳細はソース内コメントを、ファームウェア側の契約は `PONKEY_PT2` のソースを参照のこと。

姉妹文書である `PONKEY_DESIGN_PHILOSOPHY.md`(Smart Response 版)が「因果関係のある応答」を主題としたのに対し、本アプリはそこから一歩進み、**「期待の設計」=心地よさと驚きの両立**を主題とする。

## 1. コアコンセプト

### 1.1 ひとことで

> **PONKEY は、人の手に触れ返してくる楽器である。**
> SESSION は、その「触れ返し」を *音楽的な会話* にまで引き上げる試みである。

### 1.2 何を解決しようとしているか

PONKEY の核心は双方向の物理性(ソレノイドが押し返す)だが、ソフト側の「AIの返答」が古典研究レベル(模倣 or ランダム)に留まると、体験は「心地よいが驚きがない」または「驚くが意味不明」のどちらかに転ぶ。両者が**同じフレーズの中に同居**して初めて、人は「通じた」と「おっ」を同時に感じる。

本アプリのすべての設計判断は、この一点に向かっている。

### 1.3 "生きていると感じる" ための3条件

1. **あなたの言葉で話す** — 固定フレーズ集を持たず、AIは目の前の人の演奏からその場で語彙を学ぶ。
2. **期待を設計する** — ほとんどを予測可能にして安心させ、一箇所だけ狙って裏切り、必ず解決して着地する。
3. **能動性を持つ** — 受け身で待つだけでなく、ノリが高まればAIの方から暴れ出す(ノリゲージ → ソロ)。

## 2. 理論的背景

### 2.1 期待理論(Meyer / Huron)

音楽の情動は、**期待の充足と裏切りの時間的設計**から生まれる(Leonard Meyer『Emotion and Meaning in Music』1956、David Huron『Sweet Anticipation』2006)。「驚き」は心理量として、次に来る事象の**予測の外れ具合**で測れる。

本アプリはこれを情報理論の **サプライザル(surprisal)= −log₂ P(次の音)** として実装する。単位は **bit**。

- bit が小さい = 予測どおり = 心地よい・安心
- bit が大きい = 予測外 = びっくり・緊張

### 2.2 オンライン学習による続奏(Pachet: The Continuator, 2002)

François Pachet の Continuator は、演奏者から**実時間で**可変次数マルコフを学習し、「その人のスタイルで」続きを生成した。本アプリの確率モデルはこの系譜に立つ。

### 2.3 本アプリの新規性

既存研究の多くは「模倣(=驚き最小)」か「ランダム(=制御不能な驚き)」のどちらかである。生成AIも通常は尤度最大化、すなわち**驚きを最小化**する方向に働く。

本アプリは逆を行く:

> **驚きの「量」と「位置」を目標カーブとして明示的に設計し、
> 候補フレーズ群をそのカーブへの適合度で選抜する。**

これにより、びっくりは事故ではなく**意図**になり、パラメータ化・可視化・説明が可能になる。これが「びっくりカーブ」UI の理論的意味である。

## 3. システム全体像

### 3.1 三体構成

| 役 | 名前 | 役割 | 物理出力 |
|---|---|---|---|
| 人間 | PONKEY A | 演奏する。AIはここから学ぶ | A本体スピーカー or アプリ合成 |
| 旋律AI | PONKEY B | 旋律で返事する。和音・転調も | Bソレノイド + アプリ合成音 |
| ドラムAI | PONKEY C | BPM/グルーヴを検出し伴奏 | Cソレノイド(物理ビート) + ドラム音 |

すべて BLE-MIDI で個別にアプリへ接続する。アプリが「指揮者」として三者を束ねる。

### 3.2 三層アーキテクチャとの対応

PONKEY プロジェクト全体の設計原則(反射=ESP32 / 推論=ローカル / 知性=クラウド)に本アプリを当てはめると:

- **反射層(ESP32 / ファーム):** ソレノイド発火、stagger による同時発火の時分割、SAM 発音。**const テーブルで表現できる即応処理。**
- **推論層(ローカル / ブラウザ):** 本アプリの中心。確率モデル・サプライザル整形・ビート検出。**動的状態とリアルタイム計算。** 1回の作曲が約7ms で、完全にローカル完結・オフライン動作。
- **知性層(クラウド / 任意):** 現状未使用。将来 Claude を「演出家」として接続する余地(§9)。

この分離により、**API キー無し・ネット無しで全機能が動く**。展示で回線が落ちても止まらない。

## 4. 確率モデル — AIが「あなたの言葉」を覚える仕組み

### 4.1 度数空間という発想

16個のキーは物理的には固定だが、アプリ内部では **「ドレミの階段の段番号」= 度数(0〜15)** として抽象的に扱う。度数の足し算・引き算が、そのまま音楽的な移調になる。

これにより:
- モデルは音高そのものでなく**音の動き(度数列・音程列)**を学べる
- 旋法(明るい/せつない)を後から差し替えても、学習した「動きのクセ」は保たれる

### 4.2 何を学ぶか

演奏が1フレーズ終わるたびに、モデルは2種類のクセを記録する:

1. **度数マルコフ(order 1〜3):** 「この音列のあとに、この音が来やすい」
2. **音程マルコフ(order 1〜2):** 「この跳び方のあとに、この跳び方が来やすい」
   → 音程は移調不変なので、**少ないデータでも別の高さに般化**できる(例:低い「ドレミ」を聴いて、高い「ソラシ」も予測できるようになる)

さらに v3.2 からは:

3. **リズムマルコフ:** 「この間(ま)のあとに、この間が来やすい」(拍クラスの連鎖)

### 4.3 コールドスタート対策(音楽的事前分布)

データがまだ無い最初の一手でも音楽的に振る舞えるよう、**事前分布**を持たせている:

- **順次進行バイアス:** 近い音へ動きやすい(跳躍は珍しい)
- **安定音吸引:** ド(主音)> ソ > ミ の順に「帰りたくなる」重み

この事前分布と学習データを、観測数による信頼度(PPM風のバックオフ)で混合する。聴けば聴くほど学習データの比重が上がり、最初は無難に、だんだんあなたらしくなる。

### 4.4 検証

- 同じモチーフを8回聴かせると、その並びのサプライザルが **2.30 bit → 0.70 bit** に低下(=覚えた)
- 別の高さで弾いた同型も **0.72 bit** に低下(=移調般化が効いている)

## 5. サプライザル整形エンジン — びっくりを設計する

### 5.1 目標カーブ

返答1フレーズに対し、サプライザルの**目標カーブ**を引く:

```
bit
 高 |            ★ <- spike(ここで1回だけ裏切る)
    |
 基 |___________/ \___ <- baseline(ほぼ予測どおりで安心)
 低 |                 \_ <- 終止(解決して着地)
    +------------------------> 音の位置
```

- **baseline:** 低め一定(安心の連続)
- **spike:** 1フレーズに1箇所だけ。`baseline + 1.5 + mood×5.0` bit
- **解決:** 終わりの2音は基調より更に低く(主音・安定音へ着地)

「びっくりの強さ」スライダー = `mood`。spike の高さを直接制御する。

### 5.2 候補生成と選抜

1. あなたのフレーズ末尾を文脈に、確率モデルから候補を **120本**サンプリング
2. 各候補の各音のサプライザルを実測し、**目標カーブとの二乗誤差**で採点(spike位置と終止に重み)
3. 接続(出だしが人間の終音に近いか)・音域(端に張り付かないか)のボーナス/ペナルティを加算
4. 最良の1本を採用

重要なのは、**spike の音はランダムに選ばれるのではなく**、「目標 bit に最も近いサプライザルを持つ音」を選ぶこと。狙った驚きであり、事故ではない。

### 5.3 「ためらい」(アゴーギク)

spike の直前に半拍の間を挿入する。Huron の言う、逸脱の前の微小な「タメ」。これがあるとびっくりが「意図」として聴こえる。

### 5.4 リズムはリサイクル

返答のリズムは、あなたが弾いたリズム(IOI を拍に量子化したもの)を流用する。

> **慣れはリズムが担い、驚きは音高(や和音・間)が担う。**

両方を同時に崩すと「事故」に聞こえる。片方を安定させることが、もう片方の逸脱を引き立てる。

## 6. 多チャンネル・サプライザル(v3.0)

### 6.1 なぜ単旋律・単一チャンネルを破るのか

v2 までは「音高」だけで驚きを作っていた。だが音楽の驚きは音高だけではない。**和音**(突然の三和音、転調)、**間**(沈黙、早口)も強力な驚きの軸である。

### 6.2 3つのチャンネル

| チャンネル | 色 | 驚きの作り方 |
|---|---|---|
| **音高 PITCH** | 橙 | 予測外の跳躍 |
| **間 TIME** | 青 | 急な「ため」(長い休符)/ 倍速バースト |
| **和音 HARMONY** | 紫 | 三和音 / クラスター / 転調 |

### 6.3 鉄則:1返答1チャンネル

> **逸脱が「意図」として聴こえるのは、他チャンネルが安定している時だけ。**(Huron)

ゆえに、1つの返答でスパイクを打つのは**1チャンネルのみ**。さらに「同じチャンネルは連発しない」抑制をかけ、会話が単調にならないようにする。

### 6.4 物理和音(単旋律の打破)

度数の3度堆積(deg, +2, +4)は、度数空間では自動的にダイアトニック三和音になる。これを同時にCソレノイドへ送ると、ファームの **stagger queue** が同時発火を時分割で吸収し、**3本のソレノイドによる物理「和音スタンプ」**が打てる。終止はトニック三和音(確率 0.55)= 物理カデンツ。

### 6.5 世界を塗り替える転調(Cメジャー固定の打破)

最大の仕掛け:

> **B が「せかいがマイナーになった」と打つと、転調は持続する。
> その後あなたが弾く同じキーの響きまで、せつなくなる。**

16キーは「ドレミの階段」だが、その**階段の色(旋法)はアプリが決めている**。Major / Dorian / Lydian / Mixolydian / Minor を切り替え可能で、B の和音スパイクの一種として「転調」が出る。これを成立させるには、A 本体の SAM を `CC_SAM_MUTE` で黙らせ、両者の声をアプリ側で合成する必要がある(設定「モードのじゆう」ON)。**B があなたの弾く世界そのものを書き換える**、これが本アプリで最も詩的な瞬間である。

## 7. ノリゲージとソロ(v3.1)— AIの能動性

### 7.1 なぜゲージか

受け身で順番を待つだけのAIは「道具」に留まる。**自分から我慢できなくなって暴れ出す**瞬間があると、AIは「生き物」に近づく。

### 7.2 仕組み

- **溜まる:** あなたが音を弾く(+)/ フレーズを交わす(+大)/ グルーヴが変異する(+)
- **冷める:** しばらく触らないと自然に下がる
- **満タン → ソロ発動**

### 7.3 ソロの構成(グイグイ)

1. **引用:** 「きみのフレーズ、もらった!」— 覚えているあなたのモチーフをまず弾く
2. **暴走:** 音高 → 間 → 和音(順不同)の**3連続スパイク**。びっくりの強さは強制的に最大近くまで、テンポも前のめり
3. **大カデンツで締め** → ゲージリセット → 「すっきりした」

**ソロ中は割り込んでも止まらない**(=グイグイ)。ただし、その間に弾かれた音もちゃんと学習している(返事はしないが、聴いて覚えている=夢中)。

「かさね」モード中に満タンになった場合は、ソロではなく**2小節の暴走フィル**に化ける。

## 8. PONKEY C — ドラマー(v3.2)

### 8.1 設計上の発見

ファームの `handleMIDI` を読むと、**入力 ch9(GMドラム)は無視され、NoteOn は `data1 < 16` でソレノイドに変換されるだけ**だった。よって「Cにドラムを鳴らせ」と命じる正しい方法は、ドラムパートを keyIndex に割り当てて**ソレノイドを直接発火させる**こと。

> **結果として C は「物理的にリズムを刻む PONKEY」になる。
> 打鍵音そのものがビートになる。** これは PONKEY の物理性と完全に整合する。

割り当て:KICK → keyIndex 12、SNARE → 14、HAT → 3。和音同様、ファームの stagger が同時発火を吸収する。

### 8.2 ビート検出(アプリ側)

A と B の**両方**の発音を時間軸に投げ込み(B は軽いアクセントで)、アンサンブル全体からノリを推定する:

- **BPM:** 全発音の IOI 中央値を「8分音符」と仮定して4分に換算、80〜180 bpm へ折り畳み、**指数移動平均で安定化**(倍/半テンポのちらつきを防ぐ)
- **グルーヴ:** 発音時刻を8分グリッドの位相(0〜7)に量子化してヒストグラム化。人がよく食う裏拍を検出すると、**そこにシンコペーションのキックを足す**

### 8.3 ドラムマップ生成

| パート | 基本配置 | 追従 |
|---|---|---|
| HAT | 8分刻み | 密度が低いと表拍だけに間引き |
| SNARE | バックビート(2・4拍目)固定 | — |
| KICK | 1・3拍目固定 | グルーヴ位相の食いに追従(シンコペーション) |

ノリゲージが高いと、小節末に**フィル**が増える。

### 8.4 自走と滑らかな乗り換え

C は A のスケジューラとは独立した自走クロックを持つ。BPM が更新されても、**次の小節頭で滑らかに乗り換える**(急な位相飛びを起こさない)。テンポを変えると、Cが小節の変わり目で追従する様子が目で見える。

### 8.5 検証

- 120 bpm の8分連打 → 120 bpm / 信頼度 100% で検出
- 90 bpm の4分 → 90 bpm
- 半拍ずれた音1つでは BPM が急変しない(EMA安定性)
- 裏拍の強い食いを検出してキックを追加(シンコペーション生成)

## 9. ファームウェアとの契約

アプリが依存するファーム側の取り決め(`PONKEY_PT2` v4.2〜v4.5):

| 項目 | 値・挙動 |
|---|---|
| SERVICE UUID | `03b80e5a-ede8-4b33-a751-6ce34ec4c700` |
| CHAR UUID | `7772e5db-3868-4112-a1a9-f2669d106bf3` |
| 送信パケット | `[0x80, 0x80, status, d1, d2]` |
| **ソレノイド発火** | NoteOn かつ `d1 < 16` → keyIndex 直送でソレノイド発火(**B/Cのモードに非依存**) |
| 同時発火 | stagger queue が時分割で吸収(和音・ドラム同時打ち) |
| A→App の音名 | テーブル値(ch0=BASS / 他=NORMAL)。アプリは折返し探索で度数化 |
| `CC_SAM_MUTE`(0x55) | A の SAM をミュート(モードのじゆう ON 時) |
| `CC_STATE_SYNC`(0x51) | 接続直後に connected フラグを立てる |
| 自動再アドバタイズ | ファームは切断時に再アドバタイズ → アプリの自動再接続と協調 |

**B の声をアプリが出す理由:** 受信した NoteOn はソレノイドを発火させるが SAM を鳴らさない設計のため、B/C の「声」はブラウザ音源で合成する(または各機の SAM をモードに応じて使う)。

## 10. 接続と堅牢性

### 10.1 3スロット BLE 管理

A / B / C を独立スロットとして管理。各スロットは書き込みを直列化し、最低 12ms 間隔を空ける(GATT busy 対策)。

### 10.2 自動再接続(v2.0.1〜)

- `wanted` フラグで「ユーザーの意思(つなぎたい/切りたい)」を切断イベントと分離
- 不意の切断 → 指数バックオフ(1s→2s→4s→8s上限)で無限リトライ
- 再接続成功時は notify 購読・STATE_SYNC・書き込みキューを自動で張り直す
- ソレノイド突入のブラウンアウトで ESP32 がリセットしても、再起動 → 再アドバタイズを待って自動復帰

展示で一日中つけっぱなしにする運用を想定している。

### 10.3 ハードウェア無しでも動く

Web Bluetooth が無い環境でも、画面のグリッドをタップ(または `qwer/asdf/zxcv/bnm,` キー)すれば、デモ入力として全機能(学習・サプライザル整形・ドラム検出・ゲージ・ソロ)を体験できる。プレゼンや初見の人への説明に有効。

## 11. 可視化 — AIの内面を見せる

本アプリは「ブラックボックスでない AI」を志向する。各 UI は AI の内部状態の窓である:

- **びっくりカーブ:** 返答1音ごとの実サプライザル(色=チャンネル)と目標カーブ(点線)。「AIが今なにを企んだか」が1枚でわかる
- **ごい:◯パターン:** 学習した語彙(クセ)の量
- **ノリゲージ:** AIの興奮度。満タンでソロ
- **ドラム譜 + BPM:** 検出したテンポとノリ、リアルタイムで光るステップ
- **会話テープ:** 各フレーズをミニピアノロールで記録。spike には枠、和音は積み重ね表示

これらは展示・助成金申請・SXSW のデモで、**「人工知能類」の概念を視覚的に語る装置**としてそのまま使える。

## 12. 設計の通底原理(まとめ)

1. **心地よさ × びっくりの両立** — すべての出発点。サプライザル目標カーブで実装。
2. **あなたの言葉で話す** — 固定フレーズを持たず、その場で学ぶ。
3. **片方を安定させ、片方で逸脱する** — リズム/音高/和音/間。同時に崩さない。
4. **驚きは意図であって事故でない** — 量と位置を設計し、可視化する。
5. **物理性との整合** — 和音もドラムもソレノイドの物理発火。打鍵音がそのまま音楽。
6. **能動性** — 待つだけでなく、ノリが高まれば自分から暴れる。
7. **世界を書き換える** — 転調が持続し、人の弾く響きまで変える。
8. **止まらない堅牢性** — 自動再接続、オフライン完結、ハード無しでも動く。
9. **ブラックボックスにしない** — AIの内面を常に可視化する。

## 13. 既知の限界と今後

### 13.1 現状の正直な限界

- **和音・グルーヴの bit は設計事前分布**であり、音高・リズムのような学習値ではない(ソース内に明記)。和声言語そのものの学習はまだ。
- **モノフォニック中心** — A の入力は単音前提の解析。同時押し(和音入力)の解釈は未対応。
- **旋法は5種・1つの主音** — 自由な調域・複雑な和声進行は扱わない。
- **ビート検出は実用版** — full のビート誘導モデル(複数仮説の動的計画法など)ではなく、IOI中央値+位相投票の最小実装。複雑なポリリズムには弱い。

### 13.2 ロードマップ候補

- **知性層(Claude)の接続:** 数往復ごとに会話履歴(ABC記法等)を読み、「次の数往復の脚本」(緊張の弧・どこで裏切るか・第N往復で冒頭モチーフを回収)を返す**長期形式の演出家**。レイテンシ非依存、キー無しでもローカル脳だけで動作する設計を保つ。
- **成長の永続化:** マルコフ木とモチーフを保存し、日をまたいで「あなたの訛り」を蓄積。「この子、覚えてる」体験。
- **多体カノン:** 同じ脳に役割ポリシー(影/対旋律/ペダル)を配り、1台のAに複数のBが時間差で輪唱 → PONKEY PARK のミニチュア。
- **和声・リズムの学習化:** 現在事前分布で与えている和音・グルーヴの bit を、観測から学習する方向へ。
